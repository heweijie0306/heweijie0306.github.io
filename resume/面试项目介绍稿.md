## 开场定位（30-40秒）

面试官好，我叫何伟杰，Penn State 电气工程硕士，研究方向是图神经网络和深度学习。毕业后加入 Pageon，一家 A-B 轮的 AI 设计 SaaS 公司，作为核心研发直接向 CEO 汇报，同时带了 2 名实习生。

我主要负责两块：底层数据架构和 Agent 系统。过去大半年最重要的一件事，是找到了一个让 AI 同时具备强视觉创造力和用户可编辑性的产品方案，直接把注册到付费的转化率从不到 3% 拉到了 5% 以上，增幅超过 150%，而且是纯靠产品体验驱动的。

我展开讲一下这两个核心项目。

---

## Situation — 市场洞察与产品决策

去年年中，AI PPT 赛道出现了一个很明确的市场空白。

用户想用 AI 做 PPT，当时面临两类产品：一类是传统设计工具比如 Gamma、Canva，编辑体验好、模板精美，但 AI 能力很弱，本质上还是模板填充；另一类是新出来的通用 AI Agent 比如 Manus、Genspark，AI 创造力很强，但生成出来的 PPT 视觉效果粗糙，而且不能编辑、不能导出，用户拿到手基本没法用。

我们做了一轮系统的竞品调研后判断：**这个 gap 就是 Pageon 下半年最大的增长机会**——做一个"AI 能力和设计品质都在线"的 PPT 产品。基于这个判断，我们放弃了像 Gamma 那样先做强编辑器的路线，转向了更能释放 AI 能力的网页交互路线。

---

## Task — 核心技术挑战

方向确定后，落到技术层面其实是一个**矛盾**：

AI 生成的内容要视觉效果好，就得给模型足够大的创作自由度；但用户要能编辑、能导出，就需要结构化和约束。怎么找到一种数据格式**同时满足这两头**，是第一个核心问题。

第二个问题是 Agent 架构。我们的产品是用户通过对话来编辑 PPT 的，这意味着 AI 必须准确理解用户意图、然后稳定地执行编辑操作。如果意图识别不准或者执行不稳定，用户体验直接就废了。

所以本质上，我需要同时解决**"AI 的创造力上限"**和**"AI 的执行稳定性下限"**这两个问题。

---

## Action — 数据架构（我做了什么、为什么这么选）

### 数据格式的选型与验证

我对比测试了三种 AI 输出格式：JSON 结构化数据、Markdown、和原生 HTML。

- **JSON**：编辑性好、结构清晰，但视觉表现力太差，AI 很难通过 JSON 生成有设计感的排版。
- **Markdown**：视觉和编辑都中等，上限不够。
- **HTML**：因为它天然就是为页面排版设计的语言，足够原子化，加上 CSS 能直接支持现代网页的设计效果，视觉上限是最高的。

关键问题在于编辑性。我验证了通过 DOM Parser 对 AI 输出的 HTML 做后处理，可以转换为基于 tiptap 的富文本结构，从而支持用户的编辑和拖拽操作。这样**一套格式就同时解决了视觉效果和编辑能力**。

我独立完成了整个方案的技术验证 Demo，向团队演示后方案被采纳，并确立为产品未来的技术路线。

---

## Action — Agent 架构（从自研到迁移的演进）

### 第一阶段：自研 Agent 底座

早期产品的 AI 生成服务是 Python 后端，我在这个基础上转到 Node.js 后重新设计并实现了 Agent 的核心架构——自研了 Tool Calling 机制和 ReAct Loop 闭环，搭建了 PPT 的生成和编辑两条核心 Pipeline。生成链路包括用户意图识别、上下文信息获取、大纲生成到流式输出；编辑链路支持对 PPT 的增删改查操作。

这个阶段遇到最大的技术难点是**编辑工具的 diff 匹配不稳定**——大模型生成的替换内容经常和原文对不上。我设计了 7 种 Search & Replace 策略，包括模糊匹配和 XML 结构匹配，显著降低了编辑失败率。但说实话，这本质上是在打补丁。

### 第二阶段：迁移到 Claude Code SDK

随着功能越来越复杂，自研框架的维护成本和稳定性瓶颈越来越明显。后来 Claude Code Agent SDK 出来后，我评估了它的原生能力——File System、Bash Tool 这些能覆盖我们大概 80% 的通用场景。

团队评估后决定迁移到 Claude Code Agent SDK。在迁移过程中，我负责的是适配层的实现：通用能力直接用 SDK 的原生 Tool，业务特有的逻辑——比如流式生成、用户意图确认、PPT 结构操作——通过 MCP 协议开发了 stdio Server 接入。同时基于 MCP 也支持了 HTTP 方式接入用户的云端数据源，比如 GitHub、Notion。

这次架构升级不只是换了个框架，而是把 Agent 从一个"能用"的状态提升到了"可扩展、可维护"的工程化状态。

---

## Result — 业务结果

数据架构重构和 Agent 体系升级上线后，**在没有额外 SEO 或广告投放的情况下，注册到付费的转化率从不到 3% 在一个月内持续提升并稳定在 5% 以上，增幅超过 150%**。

从产品能力的角度，数据架构重构还拓宽了产品的能力边界——原来只能做 PPT，重构后我们成功支持了 Poster、Infographic、Social Media 等多种平面设计场景，视觉效果和 Agent 编辑能力都超越了竞品 Gamma。

这套架构也成了后续产品迭代的基础底座，包括我后来做的 MG 视频生成项目也是在这个体系上长出来的。

## 补充：MG 视频项目（如果面试官追问第三个项目）

> 这个项目是在前面的 Agent 架构基础上做的延伸。

我们要给产品新增 AI 动画视频生成的能力。核心挑战是怎么让 Agent 在浏览器端直接生成可预览的动画代码。

我基于 PiAgent SDK 和 Vercel 的 just-bash 搭建了一套完全 Serverless 的 Web 端 Agent 运行时，零容器、零 WASM。Agent 生成 TSX 代码后，通过 Babel 动态编译和静态校验，注入 Remotion 的动画 API，就能实时预览。

另外我设计了一个 Skills 系统，把 10 类动画场景——3D、转场、运动设计这些——沉淀为领域知识文档，运行时注入 Prompt 来驱动 Agent 按场景生成代码。最后通过 AWS Lambda 实现云端渲染导出。目前还在内测阶段。

## 面试官可能的追问 & 应对要点

| 追问方向 | 应对要点 |
|---------|---------|
| 转化率提升的归因？ | 数据架构带来的视觉提升 + Agent 编辑体验增强是两个主要因素。没做严格 AB 测试来拆分贡献度，但从用户反馈看视觉升级的感知最直接。 |
| 为什么不直接用 Claude Code SDK，要先自研？ | 时间线问题——我们启动 Agent 开发时 SDK 还没发布。自研阶段积累的经验让团队迁移时能更精准地判断哪些该用原生、哪些该自定义，我在迁移中主要负责 MCP 适配层的实现。 |
| 7 种 S&R 策略具体是什么？ | 从精确匹配 → 模糊匹配 → 正则匹配 → XML 结构匹配，逐级降级。核心思路是容忍模型输出的微小差异（空格、换行、缩进），同时保证不会误匹配。 |
| 团队分工？你的角色边界？ | 数据架构重构是我独立发起和验证的，从发现问题到 Demo 到方案落地。Agent Infra 是我和另一位同事协作，我主要负责 Tool 层和架构演进决策，他负责前端交互层。 |
| 为什么选 HTML 不选 SVG？ | SVG 视觉上限也很高，但编辑性差——SVG 的 DOM 结构和富文本编辑器不兼容，后处理成本太高。HTML + CSS 是视觉和编辑的最优平衡点。 |
| MCP 具体怎么用的？ | 我们开发了 stdio 类型的 MCP Server，把业务逻辑封装成 Tool 暴露给 Claude Code SDK。同时支持 HTTP MCP 让用户接入自己的数据源。本质上 MCP 是我们业务层和 Agent 框架层之间的解耦协议。 |
