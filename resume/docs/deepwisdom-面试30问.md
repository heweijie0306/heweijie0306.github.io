# DeepWisdom / Atoms.dev 面试 30 问 & Hint

## Multi-Agent 架构（核心技术信仰）

### Q1: 你怎么看多 Agent 架构 vs 单 Agent 架构的优劣？
> **Hint**: 他们坚信"AI 不会以单体出现，而会以团队形式出现"。用你在 Pageon 的经验讲——主 Agent + 子 Agent 架构的实际收益（职责清晰、可独立迭代），同时坦诚说多 Agent 的协调成本和 token 消耗问题。

### Q2: 多 Agent 之间如何避免信息丢失和上下文冲突？
> **Hint**: Ethan Ouyang 说过"最难的部分是协调——context engineering 和系统设计"。可以讲你做 MCP 适配层时如何在 Agent 间传递结构化上下文，以及 Claude Code SDK 的 system prompt 注入机制。

### Q3: 如何控制多 Agent 系统的 token 成本？
> **Hint**: 他们用开源模型做到闭源 1/10 成本。可以讨论：按任务复杂度路由不同模型（简单意图识别用小模型，复杂生成用大模型）、上下文裁剪、缓存中间结果。

### Q4: MetaGPT 的 SOP 协作模式你怎么理解？和 ReAct Loop 有什么区别？
> **Hint**: SOP 是预定义流程驱动（PM 出 spec → 架构师设计 → 工程师实现），ReAct 是动态推理驱动。你在 Pageon 做的生成 Pipeline（意图识别→大纲→流式生成）本质上也是一种 SOP。关键是讲清楚什么时候用 SOP 什么时候用 ReAct。

### Q5: 多 Agent 系统中某个 Agent 输出质量不稳定，怎么处理错误传播？
> **Hint**: 他们有"user agent 来验证结果"和"dogfooding + HITL"机制。你可以讲自研阶段做的自动纠错和 HITL 中断设计。

### Q6: 如果让你设计一个 7 个角色的 Agent 团队（类似 Atoms），你怎么设计通信协议？
> **Hint**: 关键词是"智能体协议"——吴承霖说要"像当年互联网有 HTTP"。可以聊 MCP 作为 Tool 层解耦协议的实践经验，以及 Agent 间消息格式标准化。

## 开源与模型策略

### Q7: 你怎么看开源模型 vs 闭源模型在 Agent 场景中的表现差异？
> **Hint**: 吴承霖说"闭源模型尚未形成决定性优势"但也承认"Claude 整体效果更稳定，R1 幻觉难控制"。展示你两边都用过的真实体感，别一边倒。

### Q8: 如何用开源模型组合达到甚至超过闭源模型的效果？
> **Hint**: 他们的做法是 Race Mode（多模型并行，选最优结果）+ 任务级模型路由。可以结合你在 Pageon 多模型选型的经验讲。

### Q9: 开源对商业化有什么具体价值？不只是情怀。
> **Hint**: 吴承霖明确说过"通过开源扩大市场份额，为商业化奠定基础"。MetaGPT 5.8 万 stars 直接转化为 MGX/Atoms 的用户漏斗。

### Q10: DeepSeek、Qwen、Claude 各自的优缺点？你在实际项目中怎么选型？
> **Hint**: 展示你有实操判断力而不是只会用一个模型。重点讲选型的决策框架（任务类型、延迟要求、成本、稳定性）。

## 产品理念与 Vibe Coding

### Q11: 你怎么理解"Vibe Coding"到"Vibe Business"的演进？
> **Hint**: 核心是从"帮人写代码"到"帮人交付可赚钱的完整生意"。你在 Pageon 的经验——从 AI 生成 PPT 到端到端的 Agent 编辑+导出+多场景支持——本质上也是类似的产品演进逻辑。

### Q12: Atoms 说自己"不是工具，是创业拍档"，你觉得这两者的产品设计有什么本质区别？
> **Hint**: 工具等待指令，拍档主动规划。区别在于 Agent 是否有自主决策循环（research→plan→execute→iterate）。可以用 Ethan 的原话。

### Q13: 如何衡量一个 AI Agent 产品的"端到端交付质量"？
> **Hint**: 不只是代码能跑，还要能部署、能支付、能被用户使用。讲你在 Pageon 怎么定义"生成质量"——视觉效果+可编辑性+可导出性，而不只是"内容正确"。

### Q14: 面向非技术用户的 Agent 产品，最大的设计挑战是什么？
> **Hint**: Ethan 说"用户不需要有很清晰的想法，只需要做关键决策"。核心挑战是意图澄清——怎么从模糊输入提取可执行 spec。你做的多轮意图识别可以直接映射。

### Q15: 你觉得 AI 编程工具的终局形态是什么？
> **Hint**: 吴承霖说"我们的目标是自然语言编程"，"编程的核心技能将变为结构化思维"。可以结合你的判断讲，不用完全同意。

## 公司文化与团队

### Q16: 你怎么理解"没有 Ego"在技术团队中的含义？
> **Hint**: 这是吴承霖招人的第一标准。不是说没有主见，而是能接受自己的方案被推翻。可以举你在 Pageon 独立发起方案、demo 被评审、最终团队决策的过程。

### Q17: 你遇到过技术方案被团队否决的情况吗？怎么处理的？
> **Hint**: 他们重视 critical thinking。展示你能"disagree and commit"——Agent 迁移不是你推动的但你全力配合实现了适配层。

### Q18: 在一个极度扁平、没有固定 Title 的团队里，你怎么推动一个技术决策？
> **Hint**: 用数据和 demo 说话。你独立做 HTML 方案验证 demo 说服团队的经历就是最好的例子。

### Q19: 你怎么带实习生？
> **Hint**: 他们团队小（核心<15 人），带人能力很重要。讲你带 2 个实习生的具体做法——给明确任务边界 vs 给探索空间。

### Q20: 你平时怎么跟进 AI 领域的最新进展？
> **Hint**: 吴承霖说自己"审阅约 20 万篇 arXiv 论文，精读 2100 篇"。不需要这么夸张，但要展示你有系统性的信息获取习惯，而不是只刷推特。

## 技术深度

### Q21: Agent 的记忆机制应该怎么设计？
> **Hint**: 吴承霖坦言"语言模型的记忆机制是基础缺陷"，他们的方案是"放弃传统向量检索，改用小模型扫上下文 + 主动记忆管理"。你可以讲 Pageon Agent 的上下文管理经验。

### Q22: 你设计的 7 种 Search & Replace 策略，具体是怎么从精确匹配降级到模糊匹配的？
> **Hint**: 这是展示工程深度的好机会。逐级讲：精确 → 忽略空白 → 正则 → XML 结构匹配 → 行级模糊。关键是讲设计思路而不是罗列。

### Q23: Tool Calling 的错误处理和重试策略怎么设计？
> **Hint**: 展示你的 ReAct Loop 里不只有 happy path。讲 HITL 中断、自动纠错、失败回退的具体设计。

### Q24: MCP 协议的优势和局限性？你在实际使用中遇到过什么坑？
> **Hint**: 他们推崇"智能体协议"标准化。讲 MCP 作为解耦层的好处（业务逻辑和 Agent 框架分离），也讲 stdio vs HTTP 的取舍和实际问题。

### Q25: 如何评估一个 Agent 系统的鲁棒性？用什么指标？
> **Hint**: 不只是"能不能跑通"，而是失败率、恢复率、平均重试次数、用户满意度。讲你在 Pageon 怎么衡量编辑成功率的。

## 市场与竞争认知

### Q26: Atoms 和 Cursor 的本质区别是什么？
> **Hint**: 面向不同用户。Cursor 是给开发者的 AI IDE，Atoms 是给非技术人员的创业平台。一个优化编码效率，一个交付商业结果。

### Q27: 你怎么看 Manus 和 Atoms 的竞争关系？
> **Hint**: 吴承霖评价 Manus"产品体验不错，但未实现技术突破，核心逻辑代表行业共识"。DeepWisdom 3 小时复现了 OpenManus。可以客观分析两者路线差异——通用 Agent vs 垂直 Agent 平台。

### Q28: "超级个体"这个趋势你怎么看？是真实需求还是概念炒作？
> **Hint**: 用数据支撑——40% 中小企业是独立运营者，Atoms 首月 50 万注册用户验证了需求真实性。但也可以讲局限性（复杂业务仍需团队协作）。

### Q29: 如果竞品也都用了多 Agent + 开源模型，Atoms 的护城河在哪？
> **Hint**: 三层：① 开源社区生态（15 万 stars 的 Foundation Agents）② 端到端商业闭环（不只生成代码，还有支付/部署/SEO）③ 数据飞轮（用户使用越多，Agent 质量越高）。

### Q30: 你为什么想加入 DeepWisdom / Atoms？
> **Hint**: 结合三点：① 技术匹配（你做过完整的 Agent 架构从自研到 SDK 演进，直接对口多 Agent 系统）② 产品理念共鸣（你在 Pageon 做的 Skills 系统和 Atoms 的 Agent 角色分工逻辑一致）③ 文化契合（你有独立发起方案的主动性，也有参与团队决策的协作能力——"没有 Ego"的实际体现）。
